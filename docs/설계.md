### 관계 요약

| From | To | 카디널리티 | 방향 |
|------|----|-----------|------|
| routes → bus_dispatches | 1:N | 단방향 (BusDispatch → Route) |
| routes → route_stops | 1:N | 단방향 (RouteStop → Route) |
| stops → route_stops | 1:N | 단방향 (RouteStop → Stop) |
| buses → bus_dispatches | 1:N | 단방향 (BusDispatch → Bus) |
| buses → gps_locations | 1:N | 단방향 (GpsLocation → Bus) |
| buses → events | 1:N | 단방향 (Event → Bus) |
| routes → events | 1:N | 단방향 (Event → Route) |
| buses → stops (current/next) | N:1 | 단방향 (Bus → Stop) × 2 |

모든 관계 단방향. `@OneToMany` 없음.

## 2. 시스템 구조

### 전체 구성

```
Browser
  │  HTTP GET (7초 폴링)
  ▼
Nginx :80
  ├── /        → Vue 3 SPA (정적 빌드)
  └── /api/    → Spring Boot :8080
                     │ JPA / HikariCP
                     ▼
                PostgreSQL :5432
                     buses · gps_locations · events
                     routes · stops · route_stops · bus_dispatches
```

### 버스 위치 갱신 흐름 (7초 tick)

```
BusSimulationScheduler.tick()
  ├─ ActiveDispatchCache.getAll()           ← 인메모리 (DB 조회 없음)
  └─ (버스마다) BusTickProcessor.process()  ← REQUIRES_NEW 트랜잭션 (버스별 격리)
            ├─ RouteStopCache               ← 인메모리 O(1) 조회 (DB 조회 없음)
            ├─ gps_locations INSERT         ← GPS 이력 기록
            ├─ buses UPDATE                 ← 현재 상태 비정규화 갱신
            └─ events INSERT (조건부)       ← 이벤트 감지 시만
```

tick 1회당 DB 왕복은 GPS INSERT + buses UPDATE 2건으로 고정된다.
노선·정류장 조회는 캐시를 사용하므로 버스 수가 늘어도 DB 부하에 영향을 주지 않는다.

### 관제 화면 조회 흐름

```
Browser (7초마다)
  → GET /api/buses
       buses LEFT JOIN bus_dispatches LEFT JOIN routes   쿼리 1회
       current_latitude 등 비정규화 컬럼으로 현재 상태 즉시 반환
       GPS 이력 테이블 스캔 없음

  → GET /api/buses/{id}/path
       gps_locations WHERE bus_id = ?   인덱스 스캔 (bus_id, recorded_at DESC)
       최근 50건만 조회
```

### GPS 이력 정리

```
DataRetentionScheduler (매일 03:00)
  → gps_locations 24시간 초과 레코드를 1,000건씩 chunk DELETE
  → 테이블 무한 증가 방지, 락 범위 최소화
```

### 시뮬레이터(MVP) vs 실제 운영 연동

MVP에서는 실제 버스 데이터가 없으므로 서버가 직접 버스 상태를 시뮬레이션한다.
실제 서비스로 전환할 때 변경이 필요한 지점은 아래와 같다.

#### 현재(MVP): 서버 주도 시뮬레이션

```
BusSimulationScheduler (7초마다 서버에서 실행)
  └─ BusTickProcessor
       ├─ 버스 위치: 정류장 간 선형 보간 (가상 이동)
       ├─ 속도:      정류장 도착 시 0~9 km/h, 구간 이동 중 20~60 km/h 랜덤
       ├─ 이벤트 감지: 랜덤 속도 변화 → SpeedBasedEventDetector (0.3G 임계값)
       └─ 충격 이벤트: 1/120 확률로 랜덤 발생
```

#### 실제 운영: 버스 단말기 → 서버 수신

```
버스 단말기 (GPS + 가속도계)
  │  실시간 전송 (MQTT / WebSocket / HTTP POST)
  ▼
서버 수신 엔드포인트
  ├─ 실제 GPS 좌표 (위도, 경도, 속도, 방향각)
  ├─ 가속도 센서값 (G-force) → ImpactEventDetector 직접 전달
  └─ SpeedBasedEventDetector: 실측 속도 변화 기반 이벤트 감지
```

#### 전환 시 변경 범위

| 항목 | MVP(현재) | 실제 운영 |
|------|-----------|-----------|
| 데이터 출처 | `BusSimulationScheduler` (서버 내부 생성) | 버스 단말기 Push |
| 위치 | 정류장 간 선형 보간 (가상) | 단말기 실측 GPS 좌표 |
| 속도 | 랜덤 (20~60 km/h) | 단말기 실측값 |
| 이벤트 감지 | 랜덤 속도 + 1/120 충격 확률 | 실측 가속도 센서값 |
| 수신 방식 | 없음 (서버가 직접 생성) | `POST /api/gps` 또는 MQTT |
| 제거 대상 | — | `BusSimulationScheduler`, `BusTickProcessor` |
| 유지 대상 | — | `SpeedBasedEventDetector`, `ImpactEventDetector`, `EventRepository` 등 감지 로직 전체 |

이벤트 감지 알고리즘(`SpeedBasedEventDetector`, `ImpactEventDetector`)과 DB 저장 구조는 MVP와 실제 운영 모두 동일하게 사용할 수 있다. 전환 시 데이터 수신 레이어만 교체하면 된다.

## 3. 설계 근거

### 3-1. N+1 방지

**버스 목록 조회** 필요 데이터: `버스번호, 노선명, 현재속도, 현재상태`

```sql
SELECT b.*, r.route_name
FROM buses b
JOIN bus_dispatches d ON d.bus_id = b.id
                     AND d.operation_ended_at IS NULL
JOIN routes r ON d.route_id = r.id
-- 쿼리 1번. N+1 없음.
```

**버스 상세 조회** 추가 필요 데이터: `현재 정류장, 다음 정류장`

| 대안 | 문제점 |
|------|--------|
| GPS 좌표로 가장 가까운 정류장 실시간 계산 | stops 풀스캔, 오차 발생, 매 조회마다 연산 |
| route_stops에서 현재 구간 계산 | GPS 기반 구간 판단 로직 복잡 |
| **buses에 current_stop_id, next_stop_id FK 직접 저장 (채택)** | LEFT JOIN 2회로 단일 쿼리 해결 |

```sql
SELECT b.*, r.route_name, cs.stop_name AS current_stop, ns.stop_name AS next_stop
FROM buses b
JOIN bus_dispatches d ON d.bus_id = b.id AND d.operation_ended_at IS NULL
JOIN routes r ON d.route_id = r.id
LEFT JOIN stops cs ON b.current_stop_id = cs.id
LEFT JOIN stops ns ON b.next_stop_id = ns.id
WHERE b.id = ?
-- 쿼리 1번. N+1 없음.
```

**지도 위 버스 위치·방향 표시**

| 대안 | 문제점 |
|------|--------|
| 조회마다 gps_locations에서 MAX(recorded_at) 서브쿼리 | 버스 N대 → 서브쿼리 N번 |
| **buses에 current_latitude, current_longitude, current_heading 비정규화 (채택)** | GPS 수신 시 같이 갱신, 조회 시 JOIN 없음 |

### 3-2. 양방향 관계 0개

JPA 양방향(`@OneToMany` ↔ `@ManyToOne`) 사용 시 발생 문제:

| 문제 | 내용 |
|------|------|
| 순환 참조 | `Route → List<Bus> → Route → ...` → JSON 직렬화 StackOverflowError |
| @JsonIgnore 남용 | 순환 차단하다 필요한 필드까지 누락 |
| 의도치 않은 로딩 | @OneToMany 기본 LAZY지만 실수로 전체 자식 목록 로드 |
| cascade 실수 | CascadeType.ALL + 양방향이면 노선 삭제 시 버스 삭제 위험 |

**확정: 모두 단방향 @ManyToOne, fetch = LAZY**

### 3-3. ONLINE/OFFLINE 상태 DB에 저장하지 않음

| 대안 | 문제점 |
|------|--------|
| buses에 status ENUM 컬럼 저장 | 서울 7,000대 × 5분마다 UPDATE = 분당 1,400건 불필요한 쓰기. 배치 실패 시 상태 불일치 |
| **last_communication_at 기준으로 조회 시점 계산 (채택)** | 컬럼 추가 없음, 항상 정확, 추가 쿼리 없음 |

```java
BusStatus status = bus.getLastCommunicationAt()
    .isAfter(LocalDateTime.now().minusMinutes(5))
    ? BusStatus.ONLINE : BusStatus.OFFLINE;
```

### 3-4. bus_dispatches 테이블 분리

서울 버스는 차량이 노선에 고정 배정되어 운행 중 노선이 바뀌는 경우는 거의 없다.
분리의 실제 이유는 **날짜별 운행 시작/종료 이력 관리**다.

`buses.operation_started_at`을 단일 컬럼으로 두면:

```
오늘 운행 시작: buses.operation_started_at = 2026-06-26 06:00
내일 운행 시작: buses.operation_started_at = 2026-06-27 06:10  ← 오늘 값 덮어씌워짐
```

어제 몇 시에 운행했는지, 총 몇 시간 운행했는지를 알 방법이 없다.

**확정: buses는 차량 마스터(물리 차량), bus_dispatches가 날짜별 운행 이력 담당**

```
buses           → 차량 고유 정보 (번호판, 현재 GPS 상태)
bus_dispatches  → 날짜별 운행 시작/종료 이력 + 배정 노선
```

**현재 운행 중인 배정 조회 기준:**

```sql
-- UNIQUE(bus_id, dispatched_date) 제약 없음
-- 오전/오후 분할 운행, 차량 교체 시 같은 날 여러 건 허용
SELECT * FROM bus_dispatches
WHERE bus_id = ?
  AND operation_ended_at IS NULL
ORDER BY operation_started_at DESC
LIMIT 1
```

**우회 노선 케이스**
국가 행사, 도로 공사 등으로 하루 정도 우회 운행이 발생할 수 있다.
GPS 좌표는 실제 우회 경로대로 계속 수집되고, 정류장 매칭이 안 되는 구간은
`current_stop_id`, `next_stop_id`가 null로 처리된다.
지도에는 버스 위치가 정상 표시되고 현재 정류장만 미확인 상태가 된다.
MVP 범위에서는 이 방식으로 커버하며, 확장 시 `route_exceptions` 테이블로 우회 구간을 별도 관리한다.

### 3-5. route_stops에 direction 추가

왕복 노선에서 상행/하행 정류장 순서가 다르고, 일방통행 구간이 있으면 정류장 자체가 달라진다.
`direction` 없이 `stop_order`만으로는 방향별 다음 정류장 계산이 불가능하다.

```sql
-- 현재 버스의 다음 정류장 계산
SELECT s.id, s.stop_name
FROM route_stops rs
JOIN stops s ON rs.stop_id = s.id
WHERE rs.route_id = :routeId
  AND rs.direction = :currentDirection    -- OUTBOUND / INBOUND 구분
  AND rs.stop_order = :currentOrder + 1
```

**direction 방향 전환 로직 (종점 도달 시):**

```
GPS 수신마다 서버에서 판단:
1. current_stop_id가 현재 direction의 마지막 정류장(max stop_order)인지 확인
2. 종점 도달 → direction 반전 (OUTBOUND → INBOUND, INBOUND → OUTBOUND)
3. 반전된 direction의 route_stops에서 stop_order = 1 정류장을 next_stop_id로 설정
4. buses.direction, next_stop_id 업데이트 (같은 트랜잭션)
```

### 3-6. events.route_id FK (route_number 스냅샷 대신)

초기 설계에서 `events.route_number VARCHAR`로 스냅샷을 두었을 때 문제:
- 노선번호는 변경되지 않으므로 스냅샷 의미 없음
- VARCHAR로 저장하면 routes 테이블과 JOIN 불가 (FK 없음)
- 노선별 이벤트 통계, 집계 쿼리 작성 불가

**확정: events.route_id BIGINT FK → routes**

이벤트 생성 시 현재 운행 중인 bus_dispatches에서 route_id를 가져와 저장.
나중에 버스 배정이 바뀌어도 이벤트 발생 당시의 노선을 정확히 가리킨다.

### 3-7. 비정규화 컬럼 dual-write 트랜잭션 보장

GPS 수신 시 두 곳을 동시에 써야 한다.

```
gps_locations에 INSERT  (이력)
buses UPDATE            (비정규화 최신값: latitude, longitude, heading, speed)
```

둘 중 하나가 실패하면 `buses.current_latitude`와 `gps_locations` 최신 레코드가 불일치한다.

**확정: 두 쓰기를 하나의 트랜잭션으로 묶음**

```java
@Transactional
public void processGpsData(GpsData data) {
    gpsLocationRepository.save(new GpsLocation(data));   // 이력 저장
    busRepository.updateCurrentLocation(data);            // 비정규화 갱신
    // 둘 중 하나 실패 시 전체 롤백
}
```

### 3-8. 운행 종료 시 buses stale 데이터 초기화

운행이 끝나도 buses 테이블의 direction, current_stop_id 등이 마지막 값으로 남는다.
다음 날 운행 전까지 어제 상태가 그대로 보이는 문제가 발생한다.

**확정: 운행 종료 처리와 buses 초기화를 같은 트랜잭션에서 처리**

```java
@Transactional
public void endOperation(Long dispatchId) {
    BusDispatch dispatch = busDispatchRepository.findById(dispatchId)...;
    dispatch.end();                          // operation_ended_at = NOW()

    Bus bus = dispatch.getBus();
    bus.resetOperationState();               // current_stop_id = null
                                             // next_stop_id = null
                                             // direction = null
                                             // current_speed = 0
}
```

## 4. 인덱스 전략

```sql
-- 현재 운행 중인 배정 조회 (버스 목록 조회 핵심)
CREATE INDEX idx_bus_dispatches_active ON bus_dispatches(bus_id, operation_ended_at)
    WHERE operation_ended_at IS NULL;

-- 날짜별 운행 이력 조회
CREATE INDEX idx_bus_dispatches_date ON bus_dispatches(dispatched_date, bus_id);

-- GPS 이력: 특정 버스의 최근 경로 조회
CREATE INDEX idx_gps_locations_bus_recorded ON gps_locations(bus_id, recorded_at DESC);

-- 이벤트: 버스별 이벤트 조회 + 시간 정렬
CREATE INDEX idx_events_bus_occurred ON events(bus_id, occurred_at DESC);

-- 이벤트: 타입별 필터링 + 시간 정렬
CREATE INDEX idx_events_type_occurred ON events(event_type, occurred_at DESC);

-- 이벤트: 심각도별 필터링 + 시간 정렬
CREATE INDEX idx_events_severity_occurred ON events(severity, occurred_at DESC);

-- 노선별 정류장 순서 조회 (next_stop 계산)
CREATE INDEX idx_route_stops_route_dir ON route_stops(route_id, direction, stop_order);
```

## 6. 향후 확장 고려

| 확장 기능 | 설계 |
|-----------|------|
| **gps_locations 파티셔닝** | MVP에서는 일반 테이블 + `(bus_id, recorded_at DESC)` 인덱스로 충분. 실제 서비스(7,000대 × 30초 = 월 6억 건)로 확장 시 PostgreSQL `PARTITION BY RANGE (recorded_at)`으로 월별 파티션 분리, 30일 초과 파티션 DROP 스케줄러 적용 |
| 실시간 영상 관제 | `camera_streams(bus_id FK, camera_type ENUM(FRONT/REAR/INNER_1/INNER_2), stream_url, is_receiving BOOLEAN, last_received_at TIMESTAMP)` — 카메라별 수신 여부를 개별 관리. buses 테이블의 단일 boolean으로 시작하지 않고, 처음부터 카메라 단위로 분리하여 "전면 카메라만 끊김" 같은 세밀한 상태 표시 가능 |
| 운전습관 분석 | EventType 확장, 집계용 `driving_scores` 테이블 |
| 사고 탐지 | severity=HIGH + IMPACT 조합 필터 |
| 유실물 탐지 | `lost_items(bus_id FK, found_at, image_url, status)` |
| 민원 자동 리포트 | `complaints(event_id FK, report_url, created_at)` |
| 승객 안전 분석 | EventType 확장 (PASSENGER_ASSAULT, PASSENGER_COLLAPSE 등) |
| 우회 노선 관리 | `route_exceptions(route_id FK, exception_date, description, detour_path)` |
