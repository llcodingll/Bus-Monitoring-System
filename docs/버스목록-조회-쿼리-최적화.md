# 버스 목록 조회 쿼리 최적화

> 작업일: 2026-06-26
> 대상: `GET /api/buses`

## 해결한 문제

### 1. 불필요한 JOIN FETCH 제거 (`BusDispatchRepository`)

`findAllActiveWithRoute()`에 `JOIN FETCH d.bus`가 있었으나 이는 불필요했다.

서비스 레이어에서 `d.getBus().getId()`만 사용하는데, Hibernate 프록시는 `@Id` 필드를 초기화 없이 반환하므로 `JOIN FETCH d.bus`를 붙이면 오히려 buses 테이블을 불필요하게 조인하게 된다.

**수정 전:**
```sql
SELECT d FROM BusDispatch d
JOIN FETCH d.route
JOIN FETCH d.bus         -- 불필요
WHERE d.operationEndedAt IS NULL
```

**수정 후:**
```sql
SELECT d FROM BusDispatch d
JOIN FETCH d.route
WHERE d.operationEndedAt IS NULL
```

### 2. 복합 인덱스 추가 (`BusDispatch`)

`bus_dispatches` 테이블에서 `WHERE operation_ended_at IS NULL` 조건 + `bus_id`, `route_id`, `direction` 조회가 반복되므로 커버링 인덱스를 추가했다.

```java
@Table(name = "bus_dispatches",
       indexes = @Index(columnList = "operation_ended_at, bus_id, route_id, direction"))
```

PostgreSQL은 이 복합 인덱스로 인덱스 온리 스캔이 가능하다.
향후 `WHERE operation_ended_at IS NULL` 전용 부분 인덱스(Flyway DDL)로 교체하면 인덱스 크기를 더 줄일 수 있다.

### 3. 코드 정리 (`SeedService`)

- `import static java.util.stream.Collectors.toMap` 적용
- 인라인 주석 전체 제거
