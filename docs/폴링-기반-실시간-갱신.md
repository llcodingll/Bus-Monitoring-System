# 폴링 기반 실시간 갱신

> 작업일: 2026-06-26
> 대상: 프론트엔드 전체, `usePolling` composable, `DashboardView`

## 해결한 문제

### 1. WebSocket 대신 HTTP 폴링 채택

버스 위치·이벤트를 실시간에 가깝게 갱신하는 방법으로 두 가지를 검토했다.

| 항목 | WebSocket | HTTP 폴링 (채택) |
|------|-----------|-----------------|
| 서버 인프라 | 커넥션 풀, 세션 유지, 스케일아웃 시 Sticky Session 또는 Redis Pub/Sub 필요 | 상태 없음, 일반 REST 서버 그대로 사용 |
| 연결 관리 | 재연결 로직, 하트비트, 오류 복구 직접 구현 필요 | 브라우저 HTTP 스택이 처리 |
| 지연 허용 범위 | 수십 ms 이하가 필요한 경우 유리 | 수 초 단위 갱신으로 충분한 경우 동등 |
| MVP 복잡도 | 높음 | 낮음 |

관제 화면에서 버스 위치는 7초 간격 시뮬레이터 주기와 동일하게 갱신되면 충분하다.
마커 애니메이션으로 시각적 실시간감을 보완하므로 폴링으로도 UX를 만족한다.

### 2. `usePolling` — 이중 실행 방지 · 언마운트 정리

폴링 로직을 composable로 추출해 컴포넌트 어디서든 재사용하고, 두 가지 리소스 누수를 방지한다.

```ts
export function usePolling(fn: () => void | Promise<void>, intervalMs: number) {
  let timer: ReturnType<typeof setInterval> | null = null

  function start(): void {
    if (timer !== null) return   // ① 이중 start 방지
    timer = setInterval(fn, intervalMs)
  }

  function stop(): void {
    if (timer !== null) {
      clearInterval(timer)
      timer = null
    }
  }

  onUnmounted(stop)  // ② 컴포넌트 언마운트 시 자동 정리

  return { start, stop }
}
```

`① if (timer !== null) return` — Vue의 반응성 watcher가 여러 번 트리거될 때 타이머가 중복 생성되는 것을 막는다.

`② onUnmounted(stop)` — 다른 라우트로 이동해 컴포넌트가 언마운트될 때 타이머를 자동으로 정리한다. 호출 측에서 별도로 `stop()`을 호출하지 않아도 된다.

### 3. `@Transactional(readOnly = true)` — 읽기 트랜잭션 최적화

버스 목록·이벤트 조회는 데이터 변경이 없으므로 읽기 전용 트랜잭션을 명시했다.

```java
@Service
@Transactional(readOnly = true)   // 서비스 전체 기본값
public class BusService { ... }

@Service
@Transactional(readOnly = true)
public class EventService { ... }
```

`readOnly = true`로 설정하면 Hibernate가 dirty checking(변경 감지)과 flush를 생략한다.
읽기 복제본(Read Replica)을 도입할 경우 Spring이 이 힌트를 보고 복제본으로 라우팅할 수 있다.
