# 시뮬레이터 인메모리 상태 캐싱

> 작업일: 2026-06-26
> 대상: `BusSimulationScheduler` — 7초 간격 @Scheduled tick

## 해결한 문제

### 1. tick마다 버스 상태 DB 조회 제거

시뮬레이터는 7초마다 각 버스의 진행 상태(현재 정류장 순서, 방향, 구간 내 진행률, 이전 속도)를 기반으로 다음 위치를 계산한다.

**캐싱 전:**
```
tick 실행 → 버스별로 buses 테이블에서 현재 stop_order, direction 조회
→ 버스 N대 × 7초마다 N번 SELECT
```

**캐싱 후 (채택):**
```java
private final Map<Long, BusSimState> stateMap = new ConcurrentHashMap<>();

@Scheduled(fixedDelay = 7000)
@Transactional
public void tick() {
    List<BusDispatch> dispatches = busDispatchRepository.findAllActiveWithBusAndStops();

    for (BusDispatch dispatch : dispatches) {
        // DB 조회 없이 메모리에서 상태 읽기
        BusSimState state = stateMap.computeIfAbsent(bus.getId(), id -> initState(dispatch));
        advanceBus(bus, dispatch, state);
    }
}
```

버스 상태(`currentStopOrder`, `direction`, `progress`, `prevSpeedKmh`)는 메모리에만 유지한다.
DB에는 시각적으로 필요한 값(위경도, 현재 정류장 이름, 속도)만 기록한다.

---

### 2. 불변 레코드로 상태 스냅샷 관리

`BusSimState`를 Java `record`로 정의해 변경 불가 스냅샷으로 관리한다.
상태 변경은 새 레코드 생성으로만 가능하고, 기존 상태를 덮어쓰지 않는다.

```java
record BusSimState(int currentStopOrder, Direction direction, double progress, Double prevSpeedKmh) {}
```

tick이 끝날 때 `stateMap.put(bus.getId(), new BusSimState(...))` 로 교체.
이전 속도(`prevSpeedKmh`)를 상태에 보관해 다음 tick에서 속도 변화량(가속도)을 계산하고,
이를 `SpeedBasedEventDetector`에 전달한다.

---

### 3. initState 최초 1회만 DB 조회

서버 재시작 시 기존 상태가 없으므로 초기화가 필요하다.
`computeIfAbsent`로 최초 1회만 DB에서 현재 정류장 순서를 조회하고, 이후부터는 캐시에서 읽는다.

```java
private BusSimState initState(BusDispatch dispatch) {
    // 최초 1회만 실행
    Optional<Integer> stopOrder = routeStopRepository
            .findStopOrderByRouteIdAndStopIdAndDirection(...);
    return new BusSimState(stopOrder.orElse(1), dispatch.getDirection(), 0.0, null);
}
```
