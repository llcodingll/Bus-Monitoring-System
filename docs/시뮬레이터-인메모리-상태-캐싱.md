# 시뮬레이터 인메모리 상태 캐싱

> 작업일: 2026-06-26
> 대상: `BusSimulationScheduler`, `ActiveDispatchCache`, `BusTickProcessor` — 7초 간격 @Scheduled tick

## 해결한 문제

### 1. tick마다 버스 배차 정보 DB 조회 제거 — `ActiveDispatchCache`

시뮬레이터는 7초마다 모든 운행 중 버스의 배차 정보(노선 ID, 방향 등)를 필요로 한다.

**캐싱 전:**
```
tick 실행 → busDispatchRepository.findAllActiveWithBusAndStops()
→ 7초마다 buses + dispatches + stops JOIN 쿼리 실행
```

**캐싱 후 (채택):**
```java
@Component
public class ActiveDispatchCache {

    @PostConstruct
    public void load() {
        dispatches = busDispatchRepository.findAllActiveWithBusAndStops().stream()
                .map(d -> new ActiveDispatchDto(d.getId(), d.getBus().getId(),
                        d.getBus().getBusNumber(), d.getRoute().getId(), d.getDirection()))
                .toList();
    }

    public List<ActiveDispatchDto> getAll() { return dispatches; }
}
```

서버 기동 시 `@PostConstruct`로 한 번만 로드해 `List<ActiveDispatchDto>`(POJO)에 보관한다.
tick에서는 `activeDispatchCache.getAll()`만 호출 — DB 조회 없음.

JPA 엔티티(`BusDispatch`)를 직접 캐싱하지 않고 POJO `record`로 변환해 영속성 컨텍스트 문제를 방지한다.
`@PostConstruct`로 로드한 엔티티는 메서드 종료 후 Detached 상태가 되어 이후 Dirty Checking이 동작하지 않기 때문이다.

---

### 2. tick마다 버스 시뮬레이션 상태 DB 조회 제거 — `stateMap`

각 버스의 진행 상태(현재 정류장 순서, 방향, 구간 내 진행률, 이전 속도)는 다음 tick을 계산하는 데 필요하지만 DB에 저장하지 않는다.

```java
private final Map<Long, BusSimState> stateMap = new ConcurrentHashMap<>();

@Scheduled(fixedDelay = 7000)
public void tick() {
    List<Callable<Void>> tasks = new ArrayList<>();

    for (ActiveDispatchDto dto : activeDispatchCache.getAll()) {  // DB 조회 없음
        if (OFFLINE_BUS_NUMBERS.contains(dto.busNumber())) continue;
        BusSimState state = stateMap.get(dto.busId());             // DB 조회 없음

        tasks.add(() -> {
            BusSimState newState = busTickProcessor.process(dto, state);
            if (newState != null) stateMap.put(dto.busId(), newState);
            return null;
        });
    }
    executor.invokeAll(tasks);
}
```

`state == null`(최초 tick)이면 `BusTickProcessor.process()` 내부에서 `initState()`를 호출한다.
이후 tick부터는 `stateMap`에서 이전 상태를 읽어 DB를 전혀 건드리지 않는다.

---

### 3. 불변 레코드로 상태 스냅샷 관리

`BusSimState`를 Java `record`로 정의해 변경 불가 스냅샷으로 관리한다.

```java
record BusSimState(int currentStopOrder, Direction direction, double progress, Double prevSpeedKmh) {}
```

상태 변경은 새 레코드 생성으로만 가능하고, 기존 상태를 덮어쓰지 않는다.
tick이 끝날 때 `stateMap.put(busId, new BusSimState(...))` 으로 교체.
`prevSpeedKmh`를 보관해 다음 tick에서 속도 변화량(가속도)을 계산하고 `SpeedBasedEventDetector`에 전달한다.

---

### 4. initState — routeStopCache 활용, 최초 1회만 DB 접근

서버 재시작 또는 최초 tick 시 상태가 없으면 현재 정류장 순서를 초기화해야 한다.
`BusTickProcessor`가 `state == null`을 감지하면 `initState()`를 호출한다.

```java
// BusTickProcessor
private BusSimState initState(Bus bus, ActiveDispatchDto dto) {
    Stop currentStop = bus.getCurrentStop();
    if (currentStop == null) return null;

    Optional<Integer> stopOrder = routeStopCache.findStopOrderByStopId(
            dto.routeId(), dto.direction(), currentStop.getId()); // 인메모리 캐시 조회

    return new BusSimState(stopOrder.orElse(1), dto.direction(), 0.0, (double) bus.getCurrentSpeed());
}
```

`routeStopCache`도 `@PostConstruct`에서 미리 로드된 인메모리 캐시다.
결과적으로 tick 루프 내에서 발생하는 DB 조회는 버스당 `busRepository.findByIdWithStops()` 1건뿐이며,
시뮬레이션 상태와 노선·정류장 정보는 DB를 전혀 경유하지 않는다.
