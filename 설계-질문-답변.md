## 질문 1 — 서울시 버스 7,000대 동시접속 시 시스템 확장 방안

단순히 배포 단위를 늘리기 전에, 단일 서버에서 처리량을 최대한 끌어올리는 것이 비용 대비 우선이다.

현재 설계가 이미 해결한 것들이 있다. 갱신 주기를 7초로 설정했기 때문에 7,000대 기준 초당 GPS INSERT는 약 1,000건으로 줄어든다. 1초 간격이었으면 7,000건/s에서 메시지 브로커 없이는 감당이 어렵다. `ActiveDispatchCache`와 `RouteStopCache`는 시뮬레이션 tick마다 발생하던 배차·정류장 DB 조회를 인메모리 HashMap으로 대체해, tick 1회당 DB 왕복을 GPS INSERT + buses UPDATE 2건으로 고정한다. `BusTickProcessor`의 `REQUIRES_NEW` 트랜잭션은 버스별로 커넥션을 분리해, 한 버스 실패가 다른 버스 트랜잭션을 오염시키지 않는다. `buses` 테이블의 비정규화 컬럼(`current_latitude`, `last_communication_at` 등)은 현재 상태 조회 시 GPS 이력 테이블 스캔을 없애고, `DataRetentionScheduler`의 24시간 TTL 청크 삭제로 `gps_locations` 테이블이 무한 증가하지 않도록 막는다.

단일 서버에서 추가로 적용할 수 있는 것은 Hibernate JDBC 배치 INSERT다. 현재 `gpsLocationRepository.save()` 는 건별 쿼리를 날리지만, `spring.jpa.properties.hibernate.jdbc.batch_size=50` 으로 설정하면 50건을 한 번의 DB 왕복으로 처리해 커넥션 점유 시간을 크게 줄인다. `BusSimulationScheduler`의 tick 루프를 고정 크기 스레드풀로 병렬화하면 7,000대를 7초 안에 순차 처리하는 부하를 분산할 수 있다. HikariCP 커넥션 풀도 기본 10개에서 CPU 코어 수 기준으로 조정한다.

이 최적화들을 다 적용하고도 단일 서버의 한계에 도달하면, 그때 WebSocket 게이트웨이와 Redis Pub/Sub로 수집·배포 경로를 분리하고 수평 확장을 검토한다. REST API 서버는 이미 무상태로 설계되어 있어 로드밸런서 뒤에 인스턴스를 추가하는 것만으로 읽기 부하는 즉시 분산 가능하다.

## 질문 2 — 버스별 CCTV 채널 수 확장 시 설계 방안

현재 MVP는 `buses` 테이블의 boolean 필드 하나로 영상 수신 여부를 표시한다. 채널이 늘어날 때 이 구조는 "전면 카메라만 끊김"처럼 채널별 상태를 구분하지 못하므로, 확장 시 `camera_streams` 테이블을 별도로 분리한다.

```sql
camera_streams (
  bus_id             BIGINT FK → buses,
  camera_type        ENUM(FRONT, REAR, INNER_1, INNER_2),
  is_receiving       BOOLEAN,
  last_received_at   TIMESTAMP
)
```

채널이 추가되어도 ENUM 타입 하나만 늘리면 되고, `buses` 테이블 스키마는 변경하지 않는다.

영상 원본은 차량 내 로컬 저장장치(VCR)에 채널별로 보관한다. 채널 수가 늘어도 중앙 서버 스토리지 비용은 거의 영향을 받지 않는다. 중앙 서버에는 기존 `events` 테이블이 이미 갖고 있는 이벤트 타입·발생 시각·위치 정보만 색인해, 특정 시간·장소의 영상이 어느 차량 어느 채널에 있는지 찾을 수 있게 한다. 관제센터에서 실시간으로 특정 차량을 확인해야 할 때만 온디맨드로 해당 채널 스트리밍을 요청한다. 상시 스트리밍은 대역폭 비용이 커서 제외한다. `SpeedBasedEventDetector`가 감지하는 급정거·`ImpactEventDetector`가 감지하는 충격처럼 `EventType.IMPACT` 또는 `SUDDEN_BRAKE` 이벤트가 발생하면, 해당 시점 전후 클립만 우선 업로드해 빠르게 확인할 수 있게 한다. 이 구조는 전송 비용을 최소화하면서 필요한 영상에는 신속히 접근할 수 있게 한다.
