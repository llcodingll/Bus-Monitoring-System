## 질문 1 — 서울시 버스 7,000대 동시접속 시 시스템 확장 방안

**문제:** 7,000대 동시 요청 시 Tomcat 스레드 풀(기본 200개) 고사와 DB 커넥션 고갈이 발생합니다. 수집 서버 분리 없이 직접 DB 갱신 시 대규모 락 경합 및 성능 저하가 직결됩니다.

**개선:** open-in-view: false로 커넥션 점유를 트랜잭션 단위로 축소하는 기초 체력을 확보한 뒤, 수집 레이어를 분리하여 Redis Streams에만 즉시 적재(In-Memory Write)함으로써 Tomcat 스레드 고사를 막고 트래픽을 완충합니다. 버스 단말기가 로컬 저장 후 전송하는 구조를 취해 서버의 Outbox INSERT 오버헤드를 제거합니다. Consumer Group을 통한 파티셔닝으로 순차 처리를 유도하되, 리밸런싱 구간의 중복 처리 예외는 `WHERE last_communication_at < ?` 형태의 시계열 멱등성 쿼리를 통해 분산락 비용 없이 DB 레벨에서 원자적으로 방어합니다.

## 질문 2 — 버스 전면·후면 블랙박스 및 내부 CCTV 영상 추가 시 설계 방안

**개선방안:** 영상 바이너리는 인코딩 특성상 해시가 무의미하므로, `{bus_id}_{시작시간}_{종료시간}` 메타데이터 조합으로 구간 보유 테이블(`video_clips`)을 구축하여 On-Demand 방식으로 전송합니다. 이벤트 발생 시 이미 존재하는 구간은 업로드를 생략하고 없는 구간만 단말기에 요청해 네트워크 및 스토리지 비용을 최적화합니다.

영상 압축 및 썸네일 생성 등 무거운 작업은 `ThreadPoolTaskExecutor`를 통해 메인 API 스레드와 격리하되, Bounded Queue와 거부 정책(`RejectedExecutionHandler`)을 명시하여 OOM(메모리 고갈)으로부터 서버를 보호합니다. 실시간 관제 알림은 아키텍처 단계를 나누어, 초기 단일 서버 환경에서는 인프라 비용이 없는 Spring ApplicationEvent + SSE 조합으로 처리하고, 향후 다중 서버 확장 시 분산 브로커(Redis Pub/Sub 등)로 전환하는 로드맵을 가져갑니다.
